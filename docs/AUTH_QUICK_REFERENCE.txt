WHAT HANDLES LOGIN?
===================

FRONTEND COMPONENTS:
  1. /artifacts/gusto-pos/src/pages/Login.tsx
     - Admin/Manager password login form
     - Uses POST /api/admin/login
     
  2. /artifacts/gusto-pos/src/components/PinPad.tsx
     - Staff PIN pad (4-digit numeric)
     - Uses POST /api/pin-login
     - Can be lock screen or initial login
     
  3. /artifacts/gusto-pos/src/components/Layout.tsx
     - Main app shell that guards all routes
     - Checks authentication via GET /api/auth/user
     - Redirects unauthenticated users to /login
     - Handles idle timeout (5 min → lock/redirect)
     
  4. /artifacts/gusto-pos/src/store.ts
     - Zustand store with:
       * language (persisted)
       * activeStaff (persisted)
       * isLocked (NOT persisted - prevents lockout)

BACKEND ENDPOINTS:
  1. POST /api/admin/login
     - Located: /artifacts/api-server/src/routes/admin-login.ts
     - Validates: username + password
     - Creates: JWT session in httpOnly cookie
     
  2. POST /api/pin-login
     - Located: /artifacts/api-server/src/routes/pin-login.ts
     - Validates: email + 4-digit PIN
     - Creates: JWT session in httpOnly cookie
     
  3. GET /api/auth/user
     - Located: /artifacts/api-server/src/routes/auth.ts
     - Returns: { isAuthenticated: bool, user: {...} }
     - Checks: JWT in cookie or Authorization header
     
  4. GET /api/auth/logout
     - Located: /artifacts/api-server/src/routes/auth.ts
     - Clears: sid cookie

CURRENT AUTHENTICATION FLOW
============================

Step 1: User navigates to /
  ↓
Step 2: Layout component fetches GET /api/auth/user
  ├─ If error → Redirect to /login
  ├─ If not authenticated → Redirect to /login
  └─ If authenticated → Show protected content
  ↓
Step 3: User not authenticated? Sees /login page
  ├─ Admin: Enters username + password → POST /api/admin/login
  └─ Staff: Enters 4-digit PIN → POST /api/pin-login (tests all users)
  ↓
Step 4: Backend creates JWT session
  ├─ Validates credentials
  ├─ Creates JWT signed with ADMIN_PASSWORD
  ├─ Sets httpOnly cookie "sid"
  └─ Returns { ok: true, user: {...} }
  ↓
Step 5: Frontend navigates to /
  ├─ Layout re-fetches GET /api/auth/user
  ├─ Cookie automatically included via credentials: include
  ├─ Backend validates JWT
  └─ Returns { isAuthenticated: true, user: {...} }
  ↓
Step 6: Syncs user to activeStaff store
  ├─ Sets language preference
  └─ Renders protected content

WHAT RECENTLY BROKE & WAS FIXED?
==================================

ISSUE #1 (April 13, 2026 - Commit 63100af):
  Problem: isLocked state was persisted to localStorage
  
  When user:
    1. Used app and got auto-locked after inactivity
    2. Closed the browser/app
    3. Reopened it
  → isLocked would still be true (persisted)
  → User would be permanently locked out
  
  Solution: Removed isLocked from Zustand persistence
  File: artifacts/gusto-pos/src/store.ts (line 36-41)
  
  Now: isLocked resets to false on every app restart
  Result: Users must re-authenticate if they restart during lock (expected)

ISSUE #2 (April 23, 2026 - Commits 4f6d1b3, 4db1d3c, afd968d):

  Causes:

    2. Layout.tsx passed wrong options to useGetCurrentAuthUser()
    3. Missing type casts in custom-fetch.ts
  
  Solution:
    1. Rewrote api-stub.ts with proper TypeScript types
    2. Fixed Layout.tsx hook usage
    3. Added proper type casts
  
  Files Modified:
    - lib/api-client-react/src/generated/api.ts (deleted)

HOW DO SESSION TOKENS WORK?
============================

JWT-based with httpOnly Cookies:

1. Creation (admin-login.ts):
   - User submits credentials
   - Backend validates
   - Creates JWT: sign({ user: {...}, createdAt, lastActivity }, ADMIN_PASSWORD)
   - JWT expires: 7 days (but has 1-hour inactivity timeout inside)
   - Sets httpOnly cookie named "sid"

2. Validation (authMiddleware.ts):
   - Every request to API
   - Middleware checks: cookie or Authorization header
   - Verifies JWT signature using ADMIN_PASSWORD
   - Checks inactivity: now - lastActivity > 1 hour → null
   - Updates lastActivity, sends new cookie

3. Frontend usage (custom-fetch.ts):
   - Every fetch() call includes: credentials: "include"
   - Browser automatically attaches sid cookie
   - No manual token management needed

SECURITY ASSESSMENT
===================

Strong Points:
  ✓ httpOnly cookies (prevents XSS)
  ✓ Stateless JWT (no server session storage)
  ✓ 1-hour inactivity timeout
  ✓ Rate limiting on login
  ✓ bcrypt password hashing
  ✓ Automatic plaintext → bcrypt migration

Weak Points:
  ✗ Hardcoded fallback password (GUSTO/0262)
  ✗ No email verification for password reset
  ✗ Very short minimum password (only checks < 4 chars fails)
  ✗ No 2FA
  ✗ No login audit trail
  ✗ Cookie theft = full access for 1 hour

WHERE TO LOOK FOR ISSUES?
===========================

If login is broken:

1. Check backend logs:
   $ grep -E "\[AdminLogin\]|\[PinLogin\]" <api-server-logs>
   
2. Check database:
   SELECT id, email, username, isActive FROM users;
   
3. Check environment:
   echo $ADMIN_PASSWORD  # Must be set for JWT secret
   
4. Check browser:
   Application → Cookies → Look for "sid" cookie
   Console → Check for fetch errors to /api/auth/user
   
5. Check frontend code:
   - Is credentials: "include" set? (Yes, in custom-fetch.ts line 361)
   - Is useGetCurrentAuthUser being called? (Yes, in Layout.tsx)
   - Is auth guard redirecting correctly? (Yes, has multiple checks)
